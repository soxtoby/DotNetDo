# Tool requirements guarantee availability, not versions

Workspace tool requirements are declared in `dotnetdo.toml` by OS-neutral canonical names (the lowercase `Tools.X` namespace name, never a platform package name) and are satisfied whenever the tool's executable is available, regardless of installation provenance or version. Installation is always explicit — `:install` or awaiting a `ToolInstall` — and never triggered by executing a tool command. A requirement may optionally map to a Scoop package. On Windows, missing mapped requirements install through Scoop, and the Scoop bootstrap follows the current official installer (opting into its official elevated mode under elevation) rather than pinning a version. A missing requirement without a package mapping fails with no configured installer.

We chose availability over pinned versions because platform installers like Scoop do not reliably support installing or holding arbitrary versions across packages, and duplicating version resolution would put DotNetDo in the package-manager business; teams needing exact versions still get them via provenance they control (containers, pinned CI images, manifest-owned .NET local tools per ADR 0012). Scoop was chosen over winget for its non-admin, user-scoped installs and predictable shim-based PATH behavior on build agents. Other operating systems fail installs with a clear unsupported-platform error until real resolutions are added.
