# Task parameters are discovered without execution

Task parameters are declared through tiny literal DotNetDo API calls that return parameter wrappers, such as `Do.Param("configuration", "Debug", "Build configuration")`, `Do.Param("feed", null, "Package feed")`, `Do.Param<bool>("pack", description: "Create package")`, or `Do.Secret("nuget-api-key").Required()`. DotNetDo discovers only that literal API subset by source scanning because executing a task in metadata mode could still cause user side effects; dynamic parameter calls may work at runtime but are not listable. Runtime values resolve through a Microsoft.Extensions.Configuration-backed pipeline in this order: long command-line options, `DOTNETDO_`-prefixed environment variables, user secrets, API default value, then `default`. DotNetDo owns typed parsing so tasks keep validation and fallback logic local without writing their own argument parser. String parameters and secrets use a non-generic overload where the second positional string is the default value; description-only calls pass `null` for the default or use the generic overload. `.Required()` throws immediately when no value exists and returns a separate required wrapper when a value exists; immediate `.Required()` calls are shown as required in task help, while later calls are conditional runtime validation. Normal parameter wrappers implicitly convert to `T?` and required normal wrappers convert to `T`; secret wrappers do not implicitly convert to string and require explicit `Unwrap()`. DotNetDo does not reject unknown task arguments; tasks that need a custom command-line interface can inspect `Do.Args` or use System.CommandLine. `do :help <task>` scans the task source and lists its discoverable parameters, optional descriptions, and immediate `.Required()` markers without executing it; secret parameters are string-only, marked as secrets, and their default values are not printed.
